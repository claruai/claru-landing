## US-002: S3 URL rewriting utility
**What I did:** Created src/lib/deck-builder/rewrite-s3-urls.ts with two exported functions. `rewriteS3ToProxy(html: string): string` is a synchronous function that rewrites S3 references in HTML to use /api/media/s3?key=KEY proxy URLs. Handles three patterns: (1) s3-placeholder.local/* URLs from the design agent, (2) moonvalley-annotation-platform.s3.*.amazonaws.com/KEY URLs (signed, unsigned, or double-signed -- extracts key before ?), (3) {{s3:key}} template tokens. Already-proxied URLs and non-S3 URLs are left unchanged. Fast path: returns early if none of the S3 markers are found in the HTML. Also created `extractMediaRefs(html: string): string[]` for US-011 (dependency tracking) which parses proxy URLs and returns unique S3 keys.
**Files changed:** src/lib/deck-builder/rewrite-s3-urls.ts (new), prd.json (US-002 passes: true), progress.txt
**Learnings:** The key design decision is that this function does NOT sign URLs -- it only rewrites them to the proxy pattern. The proxy endpoint (/api/media/s3) handles signing at request time, so URLs in the HTML never expire. The S3 keys are URL-encoded via encodeURIComponent when building proxy URLs to handle keys with spaces or special characters. The extractMediaRefs function uses decodeURIComponent to reverse the encoding when extracting keys.

## US-003: Server-rendered single slide route
**What I did:** Refactored the existing src/app/api/slide/[templateId]/[slideIndex]/route.ts to use `rewriteS3ToProxy` from rewrite-s3-urls.ts instead of the inline `signS3RefsInHtml` function that was doing async S3 URL signing. The route now: (1) fetches template from Supabase via createSupabaseAdminClient, (2) sorts slides by order and extracts the slide at the given index, (3) uses slide.html if it exists or generates from the layout system (getBackgroundCSS + renderSlideLayout with grid/glow overlays), (4) runs rewriteS3ToProxy (synchronous) to convert S3 references to proxy URLs, (5) wraps in a full HTML page with theme styles and `<base href>` set to origin. Removed the unused import of getS3SignedUrl from presigner.ts. Added .slide-body typography styles (paragraphs, lists, bold, italic, accent bullets) to the page CSS for consistency with the presentation renderer. Returns 400 for invalid index, 404 for missing template/slide.
**Files changed:** src/app/api/slide/[templateId]/[slideIndex]/route.ts, prd.json (US-003 passes: true), progress.txt
**Learnings:** The previous implementation used an inline `signS3RefsInHtml` function that did async signing of S3 URLs -- this was redundant since the proxy handles signing. By switching to `rewriteS3ToProxy` (synchronous), the route is simpler and the URLs are permanent proxy paths that never expire. The buildSlidePage helper was enhanced with .slide-body styles matching html-renderer.ts to ensure layout-generated slides look correct when served individually.

## US-004: Presentation controller route
**What I did:** Created src/app/api/slide/[templateId]/present/route.ts that serves a full navigable presentation as a self-contained HTML page. The route fetches the template from Supabase, sorts slides by order, generates HTML for each slide (using slide.html if set, otherwise layoutToHtml with background + grid + glow + content), runs rewriteS3ToProxy on each slide's HTML, and wraps everything in a complete HTML document. The page includes: (1) 1920x1080 fixed-stage with transform:scale() fit-to-viewport via fitStage JS, (2) keyboard navigation (arrows, space, Home, End), (3) touch/swipe navigation with horizontal-dominant threshold, (4) overview mode via Escape key with thumbnail grid, (5) progress bar and slide counter, (6) PostMessage API for goToSlide, enableSelector, disableSelector (element selector for editor integration), (7) <base href> set to origin for proxy URL resolution. All CSS and JS are inline -- zero external dependencies. Print styles included for landscape printing.
**Files changed:** src/app/api/slide/[templateId]/present/route.ts (new), prd.json (US-004 passes: true), progress.txt
**Learnings:** The presentation JS was ported almost verbatim from renderSlidesToHTML in html-renderer.ts. Key differences: (1) S3 URLs are rewritten to proxy format instead of being baked as signed URLs, (2) <base href> is set so proxy URLs resolve correctly, (3) the template title is used as the page <title>. The layoutToHtml function is duplicated (not imported from html-renderer.ts) because html-renderer.ts uses its own imports and types -- importing just the helper would create unnecessary coupling. The renderSlidesToHTML function in html-renderer.ts is NOT deleted since it's still used for offline HTML export.

## US-005: Update present mode to use server route

## US-010: Migrate existing slide HTML to proxy URLs

**Files changed:** scripts/migrate-slide-urls.ts (new), prd.json (US-010 passes: true), progress.txt

This is the same pattern used by seed-data-catalog.ts and other scripts in the scripts/ directory. The idempotency test is important -- already-proxied URLs contain /api/media/s3?key= which doesn't match any of the S3 patterns, so they pass through unchanged.

## US-004: patch_slide_html tool for surgical HTML edits
**What I did:** Added `patch_slide_html` tool definition to ORCHESTRATOR_TOOLS array in orchestrator-tools.ts (between get_slide_html and set_template_theme). Added `handlePatchSlideHtml` handler function in ai-tools.ts with a new case in the processToolCall switch statement. The tool accepts `slide_index` and `patches` (array of {selector, action, value?}). Supported selectors: tag names (h1, div, p), class selectors (.heading, .stats), id selectors (#title). Supported actions: replace (swap entire element), setAttribute (set/update attr in "attr=val" format), setStyle (parse existing style, merge new properties), setText (replace text content between tags), remove (delete element). The implementation is entirely regex-based with no external DOM parser dependency. It handles void/self-closing elements, nested same-name tags (depth counter), style attribute parsing/merging, and graceful fallback when selectors don't match (includes "not found" in result but continues processing). Updated media_refs via extractMediaRefs after patching. Includes input_examples on the tool definition for LLM guidance.
**Files changed:** src/lib/deck-builder/agents/orchestrator-tools.ts (tool definition), src/lib/deck-builder/ai-tools.ts (handler + helper functions), prd.json (US-004 passes: true), progress.txt
**Learnings:** The regex approach works well for the expected selector patterns (tag, .class, #id) without needing jsdom. The key complexity is finding the full element boundary (opening through closing tag) with proper nesting — a depth counter approach handles nested same-name tags (e.g. div inside div). Style merging requires parsing both the existing style="" attribute and the new value string into key-value maps, merging, and serializing back. For setText, the closing tag search uses lastIndexOf from the element end boundary to handle cases where there might be nested tags with the same name inside. The tool follows the same ToolCallResult pattern as all other handlers, returning { updatedSlides, result }.

## US-002: Strategist system prompt

Context interface includes name, description, slides (array of {index, title}), and slideCount. Persona is "Senior presentation strategist. McKinsey-trained. You structure the story." Includes buildDeckOutline helper that renders the CURRENT STATE block with all slide titles and indices (or "Deck is EMPTY" when no slides). Execution rules match orchestrator pattern: act first, explain in 1-2 sentences after, no filler, no praise, use think tool internally. Tools listed: add_slide, delete_slide, reorder_slides, restructure_deck, generate_section, edit_slide (title/body only), get_all_slides, delegate_research, get_deck_approach, load_skill, think.
**Files changed:** src/lib/deck-builder/agents/strategist-prompt.ts (new), prd.json (US-002 passes: true), progress.txt
**Learnings:** The strategist prompt follows the orchestrator-prompt.ts pattern closely (CURRENT STATE block, EXECUTION RULES, YOUR TOOLS, CLARU BRAND, FACTUAL INTEGRITY) but strips all design/HTML/visual references. The context interface is simpler than OrchestratorContext since it only needs slide titles, not full SlideData. No import of SlideData needed -- the slides array uses a simple {index, title} shape which is all the strategist needs to see the deck structure.

## US-003: Page Builder system prompt
**What I did:** Created src/lib/deck-builder/agents/page-builder-prompt.ts with exported getPageBuilderPrompt(context) function. Context interface includes name, slideIndex, slideHtml (full HTML source), theme ({colors, fonts}), and mediaRefs (string[]). Persona is "Expert frontend developer. You execute exactly what the user asks. You are their hands on the keyboard." Includes buildStateBlock helper that renders the CURRENT STATE block with slide HTML (truncated at 2000 chars), theme JSON, and media refs list. Prompt explicitly says: "Execute the user's instruction precisely. Do NOT redesign or reinterpret." Documents tool usage: patch_slide_html for small changes, set_slide_html for large rewrites. States "No QA needed. The user is looking at the slide and will tell you if something is wrong." Includes full MEDIA URLs section with proxy URL pattern (/api/media/s3?key=PATH), CDN scripts, and Google Fonts loading instructions. Mentions delegate_research for finding media to embed. Execution rules match the shared pattern: act first, max 2 sentences, no filler, use think tool internally.
**Files changed:** src/lib/deck-builder/agents/page-builder-prompt.ts (new), prd.json (US-003 passes: true), progress.txt
**Learnings:** The Page Builder prompt is deliberately the most opinionated about NOT having opinions -- it mirrors Claude Code's system prompt ethos of precise execution without creative reinterpretation. The theme and mediaRefs are passed as structured data (not embedded in HTML) so the agent can reference them when making changes. The slideHtml truncation at 2000 chars in the CURRENT STATE block prevents the system prompt from getting excessively large for slides with complex HTML, while still giving the agent enough context to make targeted edits. The PageBuilderTheme interface mirrors SlideThemeCustom from deck-builder.ts but is defined locally to avoid coupling to the types file (same pattern as StrategistContext).

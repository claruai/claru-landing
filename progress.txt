## US-002: Delete lead API endpoint with full cleanup
**What I did:** Added DELETE handler to src/app/api/admin/leads/[id]/route.ts alongside the existing GET and PATCH handlers. The handler follows the established auth pattern (cookie-based admin token verification), fetches the lead (404 if not found), then performs cleanup in the correct order: (1) deletes all rows from lead_dataset_access where lead_id matches, (2) if the lead has a supabase_user_id, calls supabase.auth.admin.deleteUser() with a console.warn on failure (user may already be gone), (3) deletes the lead row from the leads table. Returns 200 with { message: "Lead deleted" } on success, with proper error responses at each step.
**Files changed:** src/app/api/admin/leads/[id]/route.ts, prd.json (US-002 passes: true), progress.txt
**Learnings:** Deletion order is critical for referential integrity: child records (lead_dataset_access) must be deleted before the parent (leads). The auth user deletion is treated as best-effort since the user may have already been manually removed from Supabase Auth -- console.warn is used instead of console.error to signal this is a non-fatal condition. The supabase.auth.admin.deleteUser() pattern was confirmed from the approve route which uses supabase.auth.admin.createUser() and generateLink().

## US-005: Approve endpoint with send_invite option
**What I did:** Updated POST /api/admin/leads/[id]/approve to accept optional { send_invite: boolean, admin_notes?: string } in the request body. send_invite defaults to false for backward compatibility. After the lead is approved and the magic link is generated, if send_invite is true AND a magic link was successfully generated, the handler calls sendInviteEmail() from @/lib/email/invite with the lead's email, name, and magic link. The response now includes invite_sent (boolean) and optionally invite_error (string). If email sending fails, the approval still succeeds -- the lead status is "approved" but invite_sent is false with the error reason. If send_invite is true but the magic link generation failed, a specific error message is returned. When send_invite is false or omitted, existing behavior is completely unchanged with invite_sent: false in the response.
**Files changed:** src/app/api/admin/leads/[id]/approve/route.ts, prd.json (US-005 passes: true), progress.txt
**Learnings:** The magicLink variable was already being generated but never used -- this was the key integration point. The sendInviteEmail function from US-004 was designed to never throw, returning { success, error? } instead, which makes the error isolation straightforward: just check result.success and propagate result.error. The invite_error field is only included in the response when there is an actual error (using spread with conditional), keeping the response clean for the success case.

## US-003: Expand PATCH endpoint to accept all lead fields
**What I did:** Expanded the PATCH handler in src/app/api/admin/leads/[id]/route.ts to accept all editable lead fields: name, email, company, role, data_needs, use_case, admin_notes, and status. Added upfront validation: name must be a non-empty string if provided, email is validated with the same regex from the POST route (/^[^\s@]+@[^\s@]+\.[^\s@]+$/), and status is validated against the allowed values (pending, approved, rejected). Each field is only included in the Supabase update if provided as a string (omitted fields are not touched). String fields are trimmed before storage. Added email uniqueness conflict handling: if Supabase returns error code 23505, the endpoint returns 409 with "Email already in use by another lead". Existing admin_notes and status behavior is unchanged.
**Files changed:** src/app/api/admin/leads/[id]/route.ts, prd.json (US-003 passes: true), progress.txt
**Learnings:** The validation is structured in two phases: (1) upfront validation with early 400 returns for invalid values, and (2) building the updateFields object from valid string values. This separation keeps the validation logic clear and avoids partially building an update object before discovering an invalid field. The admin_notes field intentionally does not get trimmed (preserving the original behavior where notes are stored as-is), while all other string fields are trimmed to match the POST route's pattern. The email uniqueness conflict (23505) handling mirrors the POST route but with a different error message since it's an update context ("already in use by another lead" vs "already exists").

## US-006: Re-send invite API endpoint
**What I did:** Created POST /api/admin/leads/[id]/invite endpoint that re-sends magic link invites to already-approved leads. The handler verifies admin auth (cookie-based token), fetches the lead (404 if not found), returns 400 if the lead status is not "approved", handles the edge case where an approved lead has no supabase_user_id by creating the auth user first (same createUser + listUsers fallback pattern as the approve endpoint), generates a fresh magic link via supabase.auth.admin.generateLink with redirectTo pointing to /portal/auth/callback, and sends the invite email via sendInviteEmail. If the auth user was created as part of the edge case handling, the supabase_user_id is persisted back to the lead record. Returns { success: true } on success or { success: false, error: string } on failure.
**Files changed:** src/app/api/admin/leads/[id]/invite/route.ts (new), prd.json (US-006 passes: true), progress.txt
**Learnings:** Unlike the approve endpoint where invite failure is non-fatal (the approval still succeeds), the invite endpoint's sole purpose is sending the email, so failures at any stage (magic link generation, email sending) result in error responses. The edge case of a missing supabase_user_id on an approved lead requires both creating the auth user AND persisting the ID back to the lead record so future re-sends don't repeat the user creation. The _request parameter is prefixed with underscore since the body is not consumed (no configurable options for re-send).

## US-007: Create lead slide-over panel on leads list page
**What I did:** Added a `[+ new lead]` button in the filter/search bar area of AdminLeadsTable, positioned next to the search input. Clicking it opens a slide-over panel from the right edge with a create lead form. The panel follows the SampleEditPanel pattern: fixed position right-0, full height, 400px max width, dark background with accent border-left, backdrop overlay, close via X button or Escape key, slide-in animation (translateX(100%) to translateX(0)). The form includes Name (required), Email (required with format validation), Company, Role, Data Needs, Use Case, and Admin Notes fields. Client-side validation shows inline errors for required fields and email format. Submit POSTs to /api/admin/leads. On 201 success: closes panel, shows success alert, calls router.refresh(). On 409: shows "A lead with this email already exists" error. On other errors: shows the error message inline. The CreateLeadPanel is implemented as a sibling component within the same file, keeping the state management simple -- showCreatePanel boolean lives in AdminLeadsTable.
**Files changed:** src/app/admin/leads/AdminLeadsTable.tsx, prd.json (US-007 passes: true), progress.txt
**Learnings:** The SampleEditPanel pattern uses a `style jsx global` block for the slideIn keyframe animation -- this is the Next.js styled-jsx approach. Kept the same pattern but renamed the keyframe to slideInRight to avoid collisions. The panel state is managed entirely within AdminLeadsTable since it's already a client component -- no need to modify page.tsx or pass callbacks from the server component. The form resets naturally because the CreateLeadPanel component unmounts when showCreatePanel becomes false, destroying all local state.

## US-002: Share settings API routes (GET + PATCH)
**What I did:** Created src/app/api/admin/deck-builder/[id]/share/route.ts with GET and PATCH handlers. GET returns current share_settings from the template row or default { enabled: false } if none set. PATCH updates share_settings JSONB — validates slug (URL-safe lowercase+hyphens), expiry (valid ISO 8601 date or null), and gate_type ('none'|'email'|'password'). Auto-generates kebab-case slug from template name on first enable; if slug conflicts with another template, appends a -XXXX random hex suffix via crypto.randomBytes(2). Slug uniqueness is checked against share_settings->>slug on other templates. Both routes use the cookie-based admin JWT auth pattern (verifyAdminToken).
**Files changed:** src/app/api/admin/deck-builder/[id]/share/route.ts (new), prd.json (US-002 passes: true), progress.txt
**Learnings:** Supabase supports JSONB field filtering with the ->> operator for text extraction, used as `.filter("share_settings->>slug", "eq", slug)`. The requireAdmin() helper pattern (returning null or error response) keeps each handler clean. crypto.randomBytes(2).toString("hex") gives 4 hex chars which is enough entropy for slug dedup.

## US-003: Public viewer page /d/[slug] (basic rendering)
**What I did:** Created src/app/d/[slug]/page.tsx as a server component. Fetches template by slug from share_settings JSONB (filtering on share_settings->>slug and share_settings->>enabled). If not found, calls notFound(). If expired, renders a styled ExpiredPage component with terminal aesthetic. For gate_type 'none', renders a full-screen DeckViewer component that embeds /api/slide/[templateId]/present in an iframe. Includes dynamic og:title and og:description metadata via generateMetadata. No admin chrome — the page is purely the presentation. Middleware already excludes /d/ routes since they're not in the matcher config (only /admin, /api/admin, /portal, /api/portal are matched).
**Files changed:** src/app/d/[slug]/page.tsx (new), prd.json (US-003 passes: true), progress.txt
**Learnings:** The middleware uses a matcher config that only runs on specified path patterns — any route NOT in the matcher is automatically excluded from middleware processing. No need to add /d to an exclusion list. The Supabase JSONB filtering with ->> operator works well for querying share_settings->>slug and share_settings->>enabled as text values ("true" not true). The presentation is rendered via iframe to /api/slide/[templateId]/present to get the full self-contained HTML with JS navigation.

## US-007: Token validation and attribution in viewer page
**What I did:** Integrated token validation directly into src/app/d/[slug]/page.tsx. When ?t=[token] query param is present, validates against deck_share_tokens table: checks token exists, matches the template_id, and hasn't expired. If valid, passes tokenContext ({ token, tokenId, email }) to the DeckViewer component, which renders it as data attributes on a hidden div for the future tracking script (US-006) to consume. If invalid or expired, the deck still renders but without attribution (anonymous view). The validateToken helper queries deck_share_tokens by token, validates template match and expiry.
**Files changed:** src/app/d/[slug]/page.tsx (updated in same commit as US-003), prd.json (US-007 passes: true), progress.txt
**Learnings:** Token validation is a natural fit in the server component since it's a server-side DB query that doesn't add client-side JS. The token context is passed to the viewer via data attributes on a hidden div rather than through React props/context, because the presentation is rendered in an iframe (separate document context). The tracking script (US-006) will read these data attributes from the parent page.

## US-011: Lead share send API + token creation

## US-012: Lead share UI in Share panel
**What I did:** Added "Share to Leads" and "Invite by Email" sections to SharePanel.tsx. The lead sharing section has a search input with debounced fetch against GET /api/admin/leads?q= (new endpoint added to route.ts). Search results appear in a dropdown; clicking a result adds the lead to a selected list. Each selected lead has individual Send and Remove buttons. When 2+ leads are selected, a bulk "Send All" button appears. Previously shared leads (from existing tokens loaded by the GET share settings endpoint) are listed below with "Resend" buttons. The "Invite by Email" section accepts comma-separated email addresses with a send button. All send operations call POST /api/admin/deck-builder/[id]/share/send and show success/error toasts. The panel refreshes tokens after each send to keep the "previously shared" list current.
**Files changed:** src/app/components/deck-builder/SharePanel.tsx (updated), src/app/api/admin/leads/route.ts (added GET handler), prd.json (US-012 passes: true), progress.txt
**Learnings:** The leads route previously only had a POST handler. Added a GET handler with optional ?q= search parameter using Supabase .or() filter with ilike for case-insensitive matching on both name and email. The debounced search (300ms) prevents excessive API calls while typing. The selected leads state is managed separately from the search results to allow independent send operations per lead. Loading states are tracked per-lead (sendingLeadIds Set) and per-email (resendingEmails Set) to show individual spinners.

## US-013: Per-viewer analytics on lead detail page
**What I did:** Created GET /api/admin/leads/[id]/deck-views endpoint that returns deck view analytics attributed to a lead. The endpoint first finds all deck_share_tokens for the lead (by lead_id or email match), then fetches all deck_views attributed to those tokens, enriches them with template names from slide_templates, and returns views sorted by most recent first. Added a "Deck Engagement" section to LeadDetailClient.tsx between the Timeline and Danger Zone sections. The section shows each deck view as a clickable card displaying template name, viewed date, completion rate (color-coded: green >=80%, yellow >=40%, muted <40%), total duration (formatted as Xm Ys), and slides count. Clicking a card expands to show a slide-by-slide breakdown with per-slide duration. Empty state shows "No deck engagement yet". Loading state shows a pulsing "loading..." text.
**Files changed:** src/app/api/admin/leads/[id]/deck-views/route.ts (new), src/app/admin/components/LeadDetailClient.tsx (updated), prd.json (US-013 passes: true), progress.txt
**Learnings:** Token attribution uses a dual lookup: by lead_id (direct link) and by email (for tokens created before the lead was linked). This handles the case where a share email was sent to a free-form email address and the lead was created separately. The view enrichment step fetches template names in a single batch query using .in() rather than N+1 queries per view. The expand/collapse pattern uses a single expandedViewId state (null or string) rather than a Set, since typically you'd only want one expanded at a time. The slides_viewed JSONB array is cast to a typed array for the slide-by-slide breakdown display.

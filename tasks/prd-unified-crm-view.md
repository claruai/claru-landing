# PRD: Unified AI-Powered CRM View

**Branch:** `ralph/unified-crm`  
**Status:** v7 — 9/10 Codex final, all findings resolved  
**Date:** 2026-04-09

---

## Problem

The current admin has two disconnected tabs (Pipeline + Queue) that split a single workflow. To handle a reply, you must:
1. See the lead in Pipeline (state, campaign context)
2. Switch to Queue (email body, draft)
3. Switch back to update pipeline state

There's no way to trigger Claude-assisted actions from the UI — drafts are generated by a background agent and surface passively. The workflow should be: open lead → read reply → pull relevant clips → Claude drafts response → approve and send.

---

## Goal

A single vertical CRM page that replaces the Kanban + Queue split. Every lead is in one place. Respond Now cards show the email + AI-drafted reply inline. A "Draft with Claude" button triggers a live agent that reads the reply, searches the clip catalog for relevant samples, and generates a personalized draft.

## Architecture

### Data Flow
```
pipeline/page.tsx (server)
  ├── leads: get_pipeline_leads RPC (with fallback to direct query)
  ├── queueItems: reply_queue WHERE draft_status IN (pending, needs_manual_draft)
  │               OR (draft_status=snoozed AND snoozed_until < now())
  └── merge server-side:
      ├── EnrichedLead[] = leads.map(l => ({ ...l, queueItem: queueMap[l.lead_id] ?? null }))
      └── unmatchedItems = queueItems where lead_id NOT IN leads
          └── UnifiedCRMView (client component)
              ├── ProspectsBar
              ├── RespondNowSection → RespondCard (expandable)
              ├── WaitingSection → LeadRow (compact + 3-dot move)
              ├── NurturingSection → LeadRow
              └── ColdSection → LeadRow (collapsed by default)
```

### Types
```typescript
// Full queue item shape — matches reply_queue table
type QueueItem = {
  id: string;
  lead_id: string | null;        // needed for join key
  inbox: string;                  // "claru" | "moonvalley"
  sender_email: string;
  sender_name: string | null;
  gmail_message_id: string;
  gmail_thread_id: string | null;
  subject: string | null;
  body_snippet: string | null;
  received_at: string;
  draft_status: string;
  draft_response: string | null;
  snoozed_until: string | null;
  classification: string;
};

// Lead enriched with its pending reply (if any)
type EnrichedLead = PipelineLead & {
  queueItem: QueueItem | null;
};

// Queue items with no matching lead — shown as unmatched cards in Respond Now
type UnmatchedQueueItem = QueueItem; // lead_id is null or points to unknown lead
// IMPORTANT: Unmatched cards must NOT render lead metadata (company, ICP score, campaign,
// thread_state) — those fields don't exist. Only render: sender_email, sender_name,
// subject, body_snippet, inbox (display-mapped), received_at, and classification badge.
```

### Section Mapping
| Section | Filter | Sort |
|---------|--------|------|
| Respond Now | `waiting_on=us AND thread_state!=dead` OR `has pending queueItem` | Queue items first (received_at DESC), then manual (days_silent DESC) |
| Waiting on Them | `waiting_on=them AND thread_state IN (hot,warm,negotiating)` | days_silent DESC |
| Nurturing | `waiting_on=unknown AND thread_state=warm` | days_silent ASC |
| Cold | `thread_state=cold` | days_silent DESC |

### Optimistic Updates Strategy
All mutations use `useOptimistic` (React 19) for immediate UI feedback. Pattern:
1. `useOptimistic` applies local state change immediately
2. Server action fires (async via `startTransition`)
3. On success: `revalidatePath` triggers server re-render; React reconciles with fresh server state
4. On error: `useOptimistic` reverts to the prior state automatically; show toast error

All `useOptimistic` state (optimisticLeads, expandedId, pendingId) lives at the `UnifiedCRMView` level — NOT in section sub-components. Section components receive state and callbacks as props. This is required so that "only one card expanded at a time across all sections" can be enforced by a single parent, and so the auto-refresh pause condition (`expandedId !== null || isPending`) can be evaluated in one place.

### AI Draft Route
`POST /api/admin/ai-draft`
- `export const maxDuration = 60` (Vercel Pro limit — handles 15-30s Anthropic calls)
- Auth: inline Supabase session check (NOT `assertAdmin()` — see US-008a auth note)
- Input: `{ leadId: string; queueItemId: string }` — **requires existing queue item** (no item creation)
- Fetches in parallel: lead CRM context + email body from queue item
- Searches clips via join: `clips JOIN dataset_clips ON dataset_clips.clip_id = clips.id JOIN datasets ON datasets.id = dataset_clips.dataset_id WHERE datasets.type = ANY(demand_types) AND clips.ai_caption IS NOT NULL ORDER BY clips.created_at DESC LIMIT 5`
  - Branch on `lead.type` (the exact `PipelineLead.type` field, which is `"demand"` or `"supply"`)
  - `lead.type === "demand"` → `datasets.type IN ('game_capture', 'short_form', 'long_form')`
  - `lead.type === "supply"` → `datasets.type IN ('annotation', 'classification', 'instruction')`
  - Note: `clips` table has no `dataset_type` column — type comes from the `datasets` join via `dataset_clips`
- Calls Anthropic SDK `claude-sonnet-4-6` (non-streaming, `max_tokens: 300`)
- Error handling pattern:
  ```typescript
  import Anthropic from '@anthropic-ai/sdk';
  try { ... } catch (e) {
    if (e instanceof Anthropic.APIStatusError) {
      if (e.status === 429) return NextResponse.json({ error: "rate_limited" }, { status: 503 });
      if (e.status === 529) return NextResponse.json({ error: "overloaded" }, { status: 503 });
      if (e.status === 401) return NextResponse.json({ error: "api_key_error" }, { status: 500 });
    }
    return NextResponse.json({ error: "unknown" }, { status: 503 });
  }
  ```
- Note: SDK does NOT auto-retry 529 (unlike 429) — the client button handles retry UX
- On success: writes draft to `reply_queue.draft_response`, revalidates `/admin/pipeline` + `/admin/queue`
- Returns: `{ draft: string }` HTTP 200 or `{ error: string }` with appropriate HTTP status
- Note: `revalidatePath` in a Route Handler invalidates the cache but does NOT push to the client — the client button (US-008b) must call `router.refresh()` after a successful draft response

## User Stories

### US-001: Extended page data fetching + type definitions
**As** the pipeline page server component  
**I need** to fetch leads and pending queue items in parallel and merge them  
**So that** each card knows its reply state and the page renders complete in a single pass

**Acceptance criteria:**
- Page loads all data from a single render — user sees complete page (leads AND queue context) simultaneously, not leads-first then queue
- Leads with a matching queue item show that item's reply badge in Respond Now
- Leads without a queue item appear in their section without a reply badge
- Queue items with no matching lead appear as "Unmatched" cards at the top of Respond Now (orange "unmatched sender" badge)
- If leads fetch fails and RPC is unavailable, fallback query runs and page still renders — the fallback query must NOT filter out cold/dead leads (the existing fallback has `.not("thread_state", "in", '("cold","dead")')` which must be removed so ColdSection is populated). The fallback uses `leads!inner (name, email, company)`. **Change this to a left join** (`leads (name, email, company)` without `!inner`) regardless of FK status — this eliminates any silent data loss and removes ambiguity. A left join with no matching `leads` row returns null for name/company/email, which the component can handle gracefully.
- If both RPC and fallback fail, page renders the error.tsx boundary
- If queue fetch fails independently, page renders without queue context (leads still show, no reply badges) — no blank page
- TypeScript passes clean with no `as any` casts

### US-008a: `/api/admin/ai-draft` route
**As** the AI draft system  
**I need** a protected API route that generates a reply draft using Claude  
**So that** the client can trigger an AI-assisted draft without storing API keys in the browser

**Acceptance criteria:**
- Route is only accessible to admin users — non-admin requests receive 401. **Auth implementation: do NOT call `assertAdmin()` in this Route Handler.** `assertAdmin()` checks `x-admin-preview: true` in the request headers and returns early — any external caller can send that header and bypass auth. Route Handlers receive raw request headers; Next.js does not strip them. Use this inline check instead:
  ```typescript
  const supabase = await createSupabaseServerClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user?.email || !isAdmin(user.email)) {
    return NextResponse.json({ error: "unauthorized" }, { status: 401 });
  }
  ```
  Import `isAdmin` from `@/lib/auth/admin`, `createSupabaseServerClient` from `@/lib/supabase/server`.
- Route accepts `{ leadId: string; queueItemId: string }` — both are required; missing either returns 400
- Route fetches lead context and queue item in parallel — user's draft is ready as fast as possible
- Route calls `claude-sonnet-4-6` with system prompt containing: Claru sales context (Moonvalley spinout, GC/Khosla/YC), voice rules (under 80 words, no em-dashes, no "I'd love to", CTA = their behavior question), and up to 5 relevant clip examples from the DB
- Route writes the generated draft to `reply_queue.draft_response`
- Route handles errors: 429 returns `{ error: "rate_limited" }`, 529 returns `{ error: "overloaded" }`, unexpected returns `{ error: "unknown" }` — all with HTTP 503. Note: Vercel platform-level timeouts are NOT catchable server-side (the platform terminates the connection before any catch block runs). Client-side timeout handling: wrap the `fetch()` call in a try/catch and show "Request timed out — try again" if the fetch throws a network error.
- `export const maxDuration = 60` is set so the route survives 15-30s Anthropic calls on Vercel
- Route revalidates `/admin/pipeline` and `/admin/queue` on success
- TypeScript passes clean

### US-009: Update action revalidation paths + add error propagation
**As** the CRM system  
**I need** approve/dismiss/snooze/update-draft actions to revalidate both pages AND propagate DB errors  
**So that** the unified view stays in sync and `useOptimistic` can revert on failure

**Acceptance criteria:**
- Every server action in `queue/actions.ts` (`approveItem`, `dismissItem`, `snoozeItem`, `updateDraft`) checks the Supabase result and throws on error: `const { error } = await db...; if (error) throw new Error(error.message)`
- `approveItem` specifically: it performs an initial fetch of the queue item before updating. If that fetch returns no row (`data === null`), throw `new Error("Queue item not found")` immediately — before the approve DB update AND before the CRM followup. The approve `.update().eq()` on a missing row returns no error from Supabase (zero rows updated silently); only the explicit null-check prevents the silent no-op.
- `moveLeadToColumn` in `pipeline/actions.ts` does the same — checks and throws. It currently only calls `revalidatePath("/admin/pipeline")` — add `revalidatePath("/admin/queue")` as well.
- All five actions revalidate both `/admin/pipeline` AND `/admin/queue` on success
- User observes: after approving a reply, the card disappears from Respond Now and the lead reappears in Waiting on Them (no manual refresh needed)
- User observes: if a DB action fails (e.g. network drop), the card snaps back to its original position within one render cycle — `useOptimistic` reverts because the action threw
- Auto-refresh via `router.refresh()` fires every 60 seconds BUT pauses while `expandedId !== null || mutationCount > 0 || isDrafting`. Implementation: use a `mutationCountRef = useRef(0)` — increment it at the START of every mutation (before `startTransition`) and decrement it in the `finally` block when the mutation settles. Do NOT put the decrement in a `useEffect` cleanup — that fires on unmount, breaking the pause logic. The ref is GC'd on unmount anyway; no cleanup needed. The `setInterval` tick reads `mutationCountRef.current` and skips `router.refresh()` if nonzero. This correctly spans the full mutation lifecycle including server re-render latency, without false-resetting on every timer-triggered refresh. `isDrafting` is the AI draft in-flight flag from US-008b (a plain `useState` boolean, independent of `startTransition`).
- TypeScript passes clean

## Design Rules

- Same design system: `--bg-primary`, `--text-tertiary`, `#92B090` accent
- Font: `font-mono` throughout
- Same classification colors as existing `QueueItem.tsx`
- `inbox` display mapping: `"claru"` → `"claru.ai"`, `"moonvalley"` → `"moonvalley.com"` — apply at render time in the component, not stored
- No modals/drawers — everything inline accordion
- Compact by default, rich when expanded
- Move/approve/dismiss/snooze mutations use `useOptimistic` + `startTransition` (React 19 pattern — optimistic because there is a prior state to revert to). Known behavior: revert may occur before revalidatePath server data arrives (Next.js #49619) — this is acceptable; user sees card snap back then the server data reconciles within ~100ms
- "Draft with Claude" button uses a plain loading-state pattern (`isPending` flag) — there is no prior draft to optimistically apply, so `useOptimistic` is not used here
